Audit Tecnico e Funzionale - Ischia Transfer Beta

Data audit: 10/03/2026

Scope: fotografia completa della codebase attuale, senza modifiche distruttive.

1. Stack tecnologico effettivo

  • Stack principale attivo: Next.js App Router, TypeScript strict, Tailwind, Supabase, Leaflet/OSM, Zod, XLSX, parser PDF/email.
  • Stack legacy ancora presente: Express + Prisma (SQLite) + client React Vite.

2. Struttura cartelle e moduli principali

  • Core app: app/, components/, lib/, supabase/, scripts/, tests/.
  • API server in App Router: app/api/*.
  • Schema DB in supabase/migrations/* + supabase/bootstrap.sql.
  • Codice legacy separato in server/ e client/.

3. Pagine già presenti

  • Operative: dashboard, dispatch, driver, map, planning, bus-tours, inbox, services/new, hotels, analytics, pricing, onboarding, settings/whatsapp.
  • Agency: area agenzia + mie prenotazioni.
  • Supporto: login, health, share service pubblico.

4. Componenti principali esistenti

  • UI kit: DataTable, PageHeader, FilterBar, SidePanel, EmptyState, StatCard.
  • Componenti operativi: services-table, leaflet-map, timeline, export-services-button.

5. API o route server implementate

  • Import hotel OSM: /api/admin/import-hotels-ischia.
  • Inbound email/PDF: /api/inbound/email, /api/email/inbound, /api/email/import-pdf, /api/email/test-import.
  • Pricing: history, margins, matches, override.
  • Export: servizi e analytics in XLSX.
  • WhatsApp: send, webhook, settings, cron/job reminders.
  • Onboarding: tenant e driver.

6. Stato autenticazione e ruoli

  • Ruoli previsti: admin, operator, driver, agency.
  • Protezione pagine via proxy.ts su cookie ruolo.
  • Protezione API sensibili via bearer token Supabase + membership tenant.

7. Stato database / Supabase / tabelle usate

  • Multi-tenant e RLS molto estesi.
  • Tabelle core: tenants, memberships, hotels, services, assignments, status_events, inbound_emails, whatsapp_events.
  • Tabelle avanzate pricing: agencies, routes, price_lists, pricing_rules, inbound_booking_imports, service_pricing, pricing_audits.
  • Tabelle hotel OSM: hotel_aliases, hotel_import_runs.

8. Funzioni presenti per area

  • Prenotazioni: presenti, ma in parte ancora demo-store.
  • Servizi/dispatch/autisti: completi con realtime e aggiornamento stato.
  • Agenzie: area base + elementi pricing.
  • Pricing: modulo avanzato e operativo.
  • PDF/email: import manuale PDF e pipeline inbound.
  • Excel: export servizi e analytics.
  • Pagamenti: non presenti moduli online strutturati nel ramo Next.

9. Cosa è solo abbozzato

  • Agency bookings e nuova prenotazione in parte ancora dipendenti da demo mode.
  • Ingestion page più orientata a test/manuale che a flusso produzione finalizzato.

10. Cosa è rotto o incoerente

  • Coesistenza doppio stack (Next/Supabase + legacy Express/Prisma/Vite).
  • Dashboard con data "oggi" hardcoded su 2026-03-02.
  • Gating pagine via cookie ruolo non sempre allineato a sessione/ruolo Supabase.
  • Messaggi "Tenant non disponibile" quando manca membership reale in Supabase.
  • POST /api/email/test-import senza auth esplicita.
  • Inbound con fallback tenant potenzialmente ambiguo se non configurato.

11. Cosa conviene tenere

  • Modulo pricing completo.
  • Pipeline inbox/email/PDF con draft.
  • Import hotel OSM + alias + run log.
  • Export analytics/servizi e driver app realtime/offline queue.
  • Base RLS multi-tenant in Supabase.

12. Cosa conviene rifare

  • Nuova prenotazione e area agenzia in Supabase-first reale (ridurre demo-store).
  • Autorizzazione pagine allineata a sessione Supabase, non solo cookie.
  • Consolidamento endpoint inbound e regole tenant esplicite.
  • Refactor file monolitici (inbox, pricing).

13. Rischi tecnici principali

  • Debito architetturale da doppio stack nello stesso repository.
  • Rischi auth/coerenza ruolo tra UI routing e backend.
  • Rischio dati su fallback tenant inbound.
  • Rischio demo: differenze tra modalità demo e modalità Supabase reale.

Report finale per stato

Già fatto

  • Base piattaforma Next/Supabase multi-tenant con RLS.
  • Dashboard, dispatch, driver, mappa, planning, bus tours.
  • Inbox + import email/PDF + draft workflow.
  • Pricing avanzato, import hotel OSM, export Excel, WhatsApp reminders.

Parzialmente fatto

  • Flussi agenzia e prenotazioni completamente production-ready.
  • Onboarding dipendente da coerenza membership tenant.
  • Coverage test orientata soprattutto alla demo mode.

Mancante

  • Modulo pagamenti/fatturazione online nel ramo Next.
  • Hardening uniforme auth su tutte le superfici UI/API.
  • Pipeline inbound unificata senza ambiguità tenant.

Da rifattorizzare

  • Separazione netta demo mode vs production mode.
  • Refactor file ad alta complessità.
  • Allineamento bootstrap SQL, migrations e documentazione.

Da eliminare

  • Stack legacy server/ + client/ (o da separare in repo dedicato).
  • Asset/log/backups legacy nel repo applicativo principale.
Verifiche tecniche eseguite
  • pnpm lint: OK con 5 warning nel client legacy.
  • pnpm build: OK, build Next completata.
